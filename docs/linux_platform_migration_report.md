# Linux Platform Layer Migration Report

## Purpose

This report documents the migration of the Linux userspace platform layer from
the prototype under `HDMI_overlay/linux_code`. The migrated layer is deliberately
small: it maps the simple AXI-Lite peripherals that later services will use.
AXI VDMA and its framebuffers remain owned by Linux through the existing
`xilinx-vdma` provider and `/dev/hdmi-vdma` client.

## Evidence And Limits

The legacy files were inspected alongside these hardware sources:

- `HDMI_overlay.sdk/hdmi_overlay/ps7_cortexa9_0/include/xparameters.h`
- `HDMI_overlay.srcs/sources_1/bd/hdmi_in/hdmi_in.bd`
- `HDMI_overlay/linux_code/src/platform/VTC_LINUX_NOTES.md`

The repository does not retain the prompts or edit history used to produce the
prototype Linux port. The provenance below records facts visible in source files
and generated hardware metadata. It does not claim who authored each manual
adaptation or whether a particular line was generated by AI.

## Legacy File Decisions

| Legacy file | Verified role and origin | Migration decision |
|---|---|---|
| `src/platform/xparameters_linux.h` | A manually curated Linux subset of the SDK-generated `xparameters.h`. Its peripheral addresses match the Vivado block design and generated BSP metadata. | Port as `src/bsp/platform/xparameters_linux.h`, limited to the constants required by the userspace BSP. |
| `src/platform/xilinx_config_linux.c` | A manual adapter defining `XVtc_ConfigTable` for imported Xilinx VTC sources. | Do not port. `src/bsp/vtc_v7_2/src/xvtc_g.c` already defines the same table, so a second definition would conflict at link time. |
| `src/hw_platform.c` and `src/hw_platform.h` | A prototype Linux adapter that opened `/dev/mem`, mapped PL registers, and translated physical BSP addresses into Linux virtual addresses. It also retained stale direct-framebuffer, direct-VDMA, SLCR-reset, and `/proc` diagnostic concepts from earlier experiments. | Rewrite as `src/bsp/platform/linux/hw_platform.c` and `.h`, retaining only the AXI-Lite mapper and translation API. |
| `src/linux_compat.h` | A monolithic compatibility layer for building bare-metal-oriented code in Linux userspace. | Do not port. Its required concepts are already split into the focused headers under `src/bsp/bsp_compat/`. |

## Curated Hardware Parameters

The migrated `xparameters_linux.h` records physical AXI-Lite addresses. The
values below were checked against the SDK-generated `xparameters.h`.

| Peripheral | Linux constant | Physical address | SDK-generated source |
|---|---|---:|---|
| Dynamic pixel clock | `XPAR_AXI_DYNCLK_0_BASEADDR` | `0x43C00000` | `XPAR_AXI_DYNCLK_0_BASEADDR` |
| Output VTC | `XPAR_V_TC_0_BASEADDR` | `0x43C10000` | `XPAR_V_TC_0_BASEADDR` |
| Input VTC | `XPAR_V_TC_1_BASEADDR` | `0x43C20000` | `XPAR_V_TC_1_BASEADDR` |
| Overlay control | `XPAR_AXIS_VIDEO_OVERLAY_R_0_BASEADDR` | `0x43C30000` | `XPAR_AXIS_VIDEO_OVERLAY_R_0_BASEADDR` |
| Subtitle BRAM controller | `XPAR_AXI_BRAM_CTRL_0_S_AXI_BASEADDR` | `0x40000000` | `XPAR_AXI_BRAM_CTRL_0_S_AXI_BASEADDR` |
| Video GPIO | `XPAR_AXI_GPIO_VIDEO_BASEADDR` | `0x41200000` | `XPAR_AXI_GPIO_VIDEO_BASEADDR` |

The file also keeps the two VTC device IDs, VTC aliases, generator/detector
flags, GPIO metadata, and range high addresses required by the imported BSP and
future userspace drivers.

## Final Structure

```text
src/bsp/
|-- bsp_compat/                  Linux replacements for small Xilinx BSP headers
|-- platform/
|   |-- xparameters_linux.h      Reviewed hardware-specific Linux subset
|   `-- linux/
|       |-- hw_platform.h        Public AXI-Lite mapping API
|       `-- hw_platform.c        Private /dev/mem region table and lifecycle
`-- vtc_v7_2/                    Imported Xilinx VTC driver sources
```

`hw_platform_init()` opens `/dev/mem` once with `O_RDWR | O_SYNC` and maps six
userspace-owned AXI-Lite regions. `hw_platform_cleanup()` unwinds all mappings,
including partial initialization. Services can request one mapped region with
`hw_platform_base()` or translate a physical BSP address into a mapped virtual
address with `hw_platform_translate()`.

## Excluded Prototype Elements

The following legacy concepts were intentionally not migrated:

- Direct AXI VDMA register mappings
- Direct framebuffer mappings and `VIRT_TO_PHYS`
- Global framebuffer pointers
- Framebuffer `/proc/iomem` and device-tree diagnostics
- Direct SLCR mapping and optional userspace PL fabric reset
- Bare-metal UART, SCU timer, and interrupt-controller constants

VDMA buffers are allocated and mapped by `/dev/hdmi-vdma`. A userspace-triggered
global PL reset is also too broad for normal platform initialization.

## Updating The Hardware Constants

When the Vivado design changes:

1. Regenerate the SDK/Vitis BSP metadata from the updated hardware design.
2. Compare `src/bsp/platform/xparameters_linux.h` with the generated
   `xparameters.h`.
3. Update only constants for peripherals still owned by the Linux userspace BSP.
4. Recheck the device-tree ownership rules under `linux/dtsi/`.
5. Compile the BSP compatibility headers and imported VTC sources before wiring
   higher-level services.

## Board Verification

The host-side migration can be compiled without hardware. Final runtime
verification must be performed on the Zynq board as root:

1. Confirm `/dev/hdmi-vdma` exists for the separately owned VDMA path.
2. Call `hw_platform_init()` and confirm all six AXI-Lite mappings succeed.
3. Exercise `hw_platform_base()` and `hw_platform_translate()` for each mapped
   region.
4. Call `hw_platform_cleanup()` and confirm clean shutdown.
